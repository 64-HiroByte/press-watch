# PressWatch Task Handoff JA

## 目的

PressWatch の Phase タスクを小さく進め、チャットをまたいでも文脈を失わないようにする。

このスキルは上位の作業運用を扱う。ブランチの具体ルールは `phase-branch-workflow-ja`、`docs/tasks.md` の完了更新は `task-update-ja`、コミットメッセージは `commit-message-ja`、`notes/` の整理は `presswatch-notes-ja` に従う。

## 短い依頼での既定動作

ユーザーが「PressWatch の次タスクに進む」「Phase N の次タスクを始める」「次は新しいチャットで作業する」と言った場合、このスキルを使う。

明示がない場合も、次を既定で行う。

- `AGENTS.md` と関連 docs を確認する。
- Phase ブランチを確認し、最新化する。
- Phase ブランチからタスク用ブランチを切る。
- 実装後に実行可能な確認コマンドと `git diff --check` を実行する。
- 必要なら `docs/tasks.md` と関連 docs を更新する。
- 個人の学習記録として必要なら `notes/timeline/` に残す。
- 完了時に次チャット用プロンプトを提案する。

ユーザーは毎回スキル名や全手順を書かなくてよい。タスク名、Phase ブランチ、特別な制約が分かれば、その文脈から足りない確認を行う。

## 依頼プロンプト例

短く始める場合:

```text
PressWatch の Phase 3 次タスクに進みます。
Phase ブランチは `phase-3/db-persistence` です。
タスク: `press_releases` テーブルを設計する
```

引き継ぎを明示したい場合:

```text
PressWatch の次タスクを新しいチャットで進めます。
Phase ブランチは `phase-3/db-persistence` です。
直前の PR #20 は Phase ブランチへマージ済みです。

タスク:
- `press_releases` テーブルを設計する

`presswatch-task-handoff-ja` の運用に沿って、作業開始、検証、PR準備、必要な notes 記録、次チャット用プロンプト作成まで進めてください。
```

Phase 名や PR 番号は、その時点の作業に合わせて置き換える。

## 作業開始時

1. `git status --short --branch` で現在ブランチと未コミット差分を確認する。
2. ユーザーが指定した Phase ブランチを親ブランチとして扱う。
3. Phase ブランチを最新化してから、タスク用ブランチを切る。
4. 次を必要に応じて確認する。
   - `AGENTS.md`
   - `.codex/skills/phase-branch-workflow-ja/SKILL.md`
   - `docs/tasks.md`
   - タスクに関係する `docs/`
   - 直前 PR のマージ状況
   - 対象コードと既存テスト
5. `.env` や秘密情報を含みうるファイルは読まない。必要なら理由を添えて許可を得る。

## 実装中

- 要件にない機能を広げない。
- 変更はレビューしやすい粒度に保つ。
- 既存コード、既存 docs、Phase 方針との整合性を優先する。
- 依存追加が必要な場合は、必要性、代替、影響範囲を確認する。
- docs に外部公開すべき仕様や手順がある場合だけ更新する。
- 個人の理解メモや実装の振り返りは `notes/` に残す。

## 完了前チェック

- 実行可能なテスト、lint、typecheck、動作確認を実行する。
- `git diff --check` を実行する。
- 差分をセルフレビューし、バグ、セキュリティ、エラーハンドリング、設定値、docs との齟齬を確認する。
- `docs/tasks.md` は確認済みのタスクだけ完了にする。
- 変更が複数の関心事に分かれる場合は、関心事別にコミットする。

## PR 作成

- タスク PR の base は Phase ブランチにする。`main` に向けない。
- 原則 draft PR として作成する。
- PR 本文には次を短く書く。
  - このタスクで扱うこと
  - このタスクで扱わないこと
  - 確認コマンド
- PR 本文には秘密情報や具体的な認証情報を含めない。
- `DATABASE_URL` など認証情報を含みうる値は、必要がなければ概要表現にする。

## PR マージ後

1. Phase ブランチへ戻る。
2. `git pull --ff-only origin <phase-branch>` で最新化する。
3. `git status --short --branch` で作業ツリーがクリーンか確認する。
4. 必要なら次タスク用の引き継ぎプロンプトを作る。

## notes への記録

`notes/` は個人用学習記録として扱う。外部公開する必要がある内容は `docs/` に整理する。

実装後の学習メモ、設計判断、質疑で深掘りした内容、次タスクへの確認事項を残す場合は `presswatch-notes-ja` に従う。

## 次チャット用プロンプト

次タスクを新しいチャットで始める場合は、次を含むプロンプトを作る。

- 日本語で応答すること
- `AGENTS.md` と秘密情報を読まない方針
- 使用する Phase ブランチ
- 直前にマージ済みの PR
- 現在のローカル同期状態
- 次タスクの対象
- まず確認してほしいファイル
- 今回の実装方針と扱わない範囲
- 実行してほしい確認コマンド
- 最後に報告してほしいこと
