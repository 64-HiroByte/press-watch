# Task Update JA

## 概要

PressWatch の `docs/tasks.md` を更新するときのプロジェクト固有ルール。タスクのチェックは「実装した」ではなく「完了とみなせる確認が取れた」ことを表す。

## 方針

- 実装しただけではチェックを入れない
- 動作確認、テスト、レビュー観点の確認が済んだタイミングでチェックを入れる
- 未確認の作業はチェックせず、必要なら補足メモや未完了タスクとして残す
- 既知の未対応や制約がある場合は、チェック済みにせず次の作業として分ける
- Phase の実装範囲が README や外部公開用 docs に影響する場合は、Phase 完了前の「ドキュメント点検・更新」タスクを末尾に置く
- ドキュメント点検タスクは機械的に全 Phase へ足さず、起動手順、API 仕様、画面、運用手順、CI、技術前提などが変わる Phase で追加する
- `docs/tasks.md` の変更は、実装コミットとは分けることを優先する
- 誤ってタスク更新を混ぜないよう、コミット前にステージ済みファイルを確認する

## Phase 完了前のドキュメント点検

次のいずれかに当てはまる Phase では、Phase の最後に外部公開用ドキュメントの点検・更新タスクを追加する。

- README の現在の開発状況、できること、今後の予定が変わる
- `docs/local-development.md` の起動手順、確認手順、環境変数名、コマンド例が変わる
- `docs/requirements.md` の要件、MVP 範囲、後続フェーズの説明と実装状態がズレる可能性がある
- `docs/tech-stack.md`、`docs/db-migrations.md`、CI 関連 docs など、公開 docs に実装と矛盾する記述が残りうる

タスク例:

```markdown
- [ ] Phase N の完了前に外部公開用ドキュメントを現状に合わせて点検・更新する
  - [ ] `README.md` の現在の開発状況と今後の予定が Phase N の実装済み状態と一致しているか確認する
  - [ ] 関連する `docs/*.md` の手順・要件・未対応範囲が現状と矛盾していないか確認する
```

## 完了扱いの目安

- 対象機能の主要な動作を確認済み
- 関連するテスト、lint、typecheck のうち実行可能なものを確認済み
- 異常系や空データなど、要件上重要なケースを確認済み
- レビューコメントがある場合は、対応または未対応理由を整理済み
- 未確認事項が残る場合は、チェックではなく別タスクやメモとして残している

## スクレイパー実装の例

スクレイパー関連タスクは、少なくとも次を確認してからチェックを入れる。

- 保存済みHTMLまたは実HTMLでパースできる
- タイトル、日付、URLなど期待する項目が取得できる
- 欠損要素、形式不一致、異常日付で処理全体が落ちない
- 対象テストが通る

## 確認観点

- チェックは「実装済み」ではなく「確認済み」を表しているか
- `docs/tasks.md` が実装コミットに混ざっていないか
- 未確認事項を完了扱いにしていないか
