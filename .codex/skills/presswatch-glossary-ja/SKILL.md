# PressWatch Glossary JA

## 目的

PressWatch の実装や質疑で出てきた用語を、後から短時間で思い出せる索引として `notes/topics/presswatch-glossary.md` に整理する。

用語集は詳説の置き場ではなく、迷ったときに最初に見る入口として扱う。

## 使うタイミング

- ユーザーが「用語集を作る」「用語をまとめる」と言ったとき
- 作業中に新しい用語や曖昧な概念が複数出てきたとき
- 同じ用語を何度も説明しており、後から引ける形にしたいとき
- timeline / topics / usage に分散した説明への索引が必要になったとき
- 既存の用語説明が古くなり、現在のコードやメモに合わせて更新したいとき

## 保存先

- 用語集: `notes/topics/presswatch-glossary.md`

用語の詳しい説明、設計判断、使い方は次へ逃がす。

- 時系列の作業ログ: `notes/timeline/`
- 概念整理や設計判断: `notes/topics/`
- API、メソッド、コマンドの使い方: `notes/usage/`

## 基本方針

- 用語集は索引として短く保つ
- 既存項目がある場合は重複作成せず更新する
- 詳しい説明を用語集に長く書きすぎない
- 詳細が必要な場合は、既存の timeline / topics / usage へリンクする
- 対象コードがある用語は「対象ファイル」を付ける
- PressWatch での意味を優先し、一般的な定義の網羅はしない
- 秘密情報や `.env` の中身は書かない

## 追加・更新の手順

1. `notes/topics/presswatch-glossary.md` があるか確認する。
2. 既存項目に同じ用語や近い用語がないか確認する。
3. 既存メモに詳しい説明がある場合は、そのファイルを参照先にする。
4. 用語集には短い説明、PressWatch での意味、対象ファイル、詳しく見るリンクを置く。
5. `git diff --check` を実行する。

## 用語項目の形

必要な項目だけを書く。すべての用語に対象ファイルが必要なわけではない。

```markdown
## 用語

短い説明。

PressWatch での意味や今回の文脈での使いどころ。

対象ファイル:

- `path/to/file.py`

詳しく見る:

- `notes/topics/topic.md`
- `notes/usage/usage.md`
```

## 書き分け

### 用語集に書く

- 一言での意味
- PressWatch での意味
- 対象ファイル
- 詳細メモへのリンク

### topics に逃がす

- 設計判断の背景
- 迷った代替案
- 責務分担や線引きの考え方
- 後続タスクで使う判断軸

### usage に逃がす

- API、メソッド、コマンドの使い方
- 主な引数
- 返り値
- 短いコード例

### timeline に逃がす

- その日に何を実装したか
- 質疑の流れ
- 実行した確認コマンド
- 今回やらなかったこと

## 表現

- 日本語で平易に書く
- 「境界」のように抽象的で引っかかりやすい語は、必要なら「線引き」などの言い換えを併記する
- 用語の一般論より、PressWatch のコードを読むための意味を優先する
- 1項目が長くなりすぎたら、topics / usage へ分ける

## 完了前チェック

- 用語集が索引として読める長さに収まっているか
- 既存メモへのリンクが古くなっていないか
- 同じ用語を重複して追加していないか
- 対象ファイルがある用語では、どのコードの話か分かるか
- `git diff --check` を実行したか
