# PressWatch Notes JA

## 目的

PressWatch の作業で得た学び、設計判断、会話中に深掘りした論点を、後から読み返せる個人メモとして `notes/` に残す。

公開・共有すべき仕様や手順は `docs/` に整理し、個人の理解、判断理由、迷った代替案、次に見る観点は `notes/` に寄せる。

## 使うタイミング

- 実装後に、ユーザーと設計判断や学習観点の質疑が続いたとき
- 作業中に出てきた用語について質問があり、後から読み返せる解釈や説明を残したいとき
- 「なぜこの実装にしたか」「他案と比べてどうか」を後から見返したいとき
- 次タスクへ渡す注意点が、`docs/tasks.md` だけでは説明しきれないとき
- 作業直後に作ったメモを、会話後の最終メモへ更新するとき

## 保存先

- 時系列メモ: `notes/timeline/YYYY-MM-DD-topic.md`
- テーマ別メモ: `notes/topics/topic.md`

同じタスクの時系列メモがすでにある場合は、新規ファイルを増やさず既存メモを育てる。

`notes/` が Git 管理外の場合は、その前提を尊重する。勝手に Git 管理対象へ変更しない。

## 書く内容

必要に応じて次を短く整理する。

- 何を実装したか
- なぜその形にしたか
- ユーザーとの質疑で整理した論点
- 質問があった用語の解釈、今回の文脈での意味、具体例
- 迷った代替案と採用しなかった理由
- コードや docs に残した内容
- コードや docs には残さなかったが覚えておきたいこと
- 確認したコマンド
- 今回あえてやらなかったこと
- 次に続く作業と確認観点

## docs との切り分け

- `docs/`: 他者が再現・レビュー・実装判断するために必要な仕様、手順、タスク
- `notes/`: 個人の理解、学習ログ、会話で深掘りした背景、迷い、次回への申し送り

同じ内容を両方に長く重複させない。
`docs/` には結論を短く置き、`notes/` には背景や判断理由を置く。

## 書き方

- 日本語で書く
- 見出しは後から探しやすい名前にする
- 実装詳細の逐語説明より、判断理由と次に役立つ観点を優先する
- コマンドは実際に確認したものだけを書く
- 秘密情報や `.env` の中身は書かない
- 未コミット差分や進行中の調整がある場合は、その状態も分かる範囲で残す

## 完了前チェック

- 既存メモがあるタスクで重複ファイルを作っていないか
- `docs/` に残すべき仕様やタスクを `notes/` だけに閉じ込めていないか
- 個人メモに秘密情報や認証情報が入っていないか
- 次回読む人が「何を見ればよいか」「次に何を確認するか」を追えるか
