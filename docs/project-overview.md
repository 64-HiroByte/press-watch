# PressWatch - Project Overview

## 1. プロジェクト名

PressWatch

---

## 2. 背景

私は前職で、環境計量証明事業所において水質検査・大気測定・法令対応・検査体制整備などに長く携わってきた。  
その中で、環境省や厚生労働省などが公表する情報を継続的に確認し、業務への影響を把握することの重要性を強く感じていた。

一方で、現場では以下のような課題がある。

- 必要な情報を継続的かつ効率的に把握しづらい
- 関係のある分野だけを継続監視したいが、人手での確認は負担が大きい
- 情報収集が属人的になりやすく、見逃しリスクがある
- RSS未整備等の理由により、継続監視の仕組みを作りにくい

こうした課題に対し、環境省の報道発表を定期取得し、分野別・キーワード別に確認しやすくするアプリを作成する。

---

## 3. 目的

このプロジェクトの目的は、前職で培ったドメイン知識を最大限に活かし、  
**現場で本当に起こっている問題を解決するアプリを開発すること** である。

また、単なる技術学習用アプリではなく、

- ドメイン知識に基づく課題設定
- 実務を意識した設計
- 課題を整理して小さく実装する力
- AIを活用しながらも設計とレビュー責任を持つ開発スタイル

を示すプロジェクトとして位置づける。

---

## 4. 想定ユーザー

### 想定される主なユーザー

- 環境計量証明事業所の技術担当者
- 環境関連の法令・行政発表を定期的に確認する必要がある担当者
- 自分の担当分野に関係する発表を効率よく確認したい担当者

### 将来的に想定できるユーザー

- 品質保証担当者
- 管理職
- 教育担当者
- 顧客説明資料の材料を探す担当者

---

## 5. 解決したい課題

### 課題1: 情報収集の負担が大きい

環境省サイト内の報道発表には多様な分野の情報が含まれており、自分の担当業務に関係する情報を効率よく見つけるのが難しい。  
業務に直接関係しない情報まで含めて確認し続けるのは負担が大きく、次第に確認頻度が下がったり、確認自体が継続されなくなったりする。

### 課題2: 分野ごとの確認がしづらい

水質・大気・土壌・騒音・温泉など、自分の担当分野だけを追いたい場合でも、報道発表が分野横断で並ぶため効率的に確認しづらい。

### 課題3: 見逃しが発生しやすい

手作業での巡回では、担当者の忙しさや属人化により見逃しが起こりうる。

### 課題4: 後で見返しにくい

気になった発表を保存・整理し、後で見返す仕組みが弱い。

---

## 6. 提供する価値

PressWatch は、環境省の報道発表を定期取得し、  
必要な情報だけを探しやすく、見返しやすくすることで、情報収集負担の軽減を目指す。

具体的には以下の価値を提供する。

- 報道発表の一覧を一元化して閲覧できる
- カテゴリで絞り込みできる
- キーワード検索ができる
- 気になった発表を保存し、後で見返せる
- 継続監視の土台を作れる

---

## 7. MVPで実現する範囲

まずは以下をMVPとする。

- 環境省サイト内の報道発表を取得する
- 取得データをDBに保存する
- 一覧表示する
- カテゴリで絞り込みできる
- キーワード検索できる
- 気になった発表を保存できる

---

## 8. このプロジェクトでアピールしたい点

- 環境分野の実務知識に基づく課題設定
- フロントエンド / バックエンド / データ取得処理を分離した構成
- 定期実行を見据えた設計
- MVPとして適切なスコープに絞る判断
- AIを使って実装を進めつつ、人間が設計・レビュー責任を持つ開発スタイル

---

## 9. 開発方針

- 旧 Topics Checker は参考にするが、そのまま流用せず責務分割を見直す
- 実装は AI を積極的に活用する
- ただし、要件定義・設計判断・レビュー・修正指示は開発者本人が責任を持つ
- 小さく作って動かし、段階的に改善する

---

## 10. 現時点の技術選定

### 確定

- フロントエンド: Next.js (App Router), TypeScript
- バックエンド: FastAPI, Python
- データベース: PostgreSQL
- - 開発環境: Docker / docker compose
- リポジトリ構成: モノレポ

### 検討中

- Cron による定期実行
- デプロイ先

**「ドメイン知識を持つエンジニアが、現場課題を抽出し、技術で解決するプロダクト」**
として位置づける。
