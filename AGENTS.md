# リポジトリ指示

## プロジェクト概要

- このリポジトリは、環境省の報道発表を収集・整理・検索しやすくする個人開発プロジェクト **PressWatch** です。
- 前職で培ったドメイン知識を活かし、現場で実際に起こる情報収集負担や見逃しリスクを軽減することを目的とします。
- 可読性、保守性、安全性を優先し、過度に複雑な実装は避けてください。
- MVP 段階を意識し、価値の高い機能から小さく改善してください。

## 技術スタック

- フロントエンドは **Next.js 16系 + React + TypeScript** を使用します。
- フロントエンドは **App Router 構成** です。
- バックエンドは **FastAPI + Python 3.14** を使用します。
- Python の依存管理には **uv** を使用します。
- フロントエンドのパッケージマネージャーには **pnpm** を使用します。
- データベースには **PostgreSQL 18** を使用します。
- 開発環境には **Docker / docker compose** を使用します。
- リポジトリは **モノレポ構成** です。
- 既存コードの命名、責務分割、ディレクトリ構成との整合性を優先してください。

## リポジトリ構成

- `apps/web`: Next.js アプリケーション
- `apps/api`: FastAPI アプリケーション
- `packages/scraper`: スクレイピング処理
- `docs`: プロジェクトドキュメント
- `infra`: Docker 関連設定や補助ファイル

## 実装方針

- TypeScript では型安全性を重視してください。
- Python でも型ヒントを活用し、読みやすさと保守性を意識してください。
- API やスクレイピング処理では失敗時の挙動を明確にしてください。
- 外部入力や取得データは常に不正・欠損・形式揺れの可能性を考慮してください。
- エラーを握りつぶさず、根本原因を修正することを優先してください。
- 変更は小さくレビューしやすい単位に分けてください。
- 要件にない機能を勝手に広げないでください。
- 不要な抽象化や、MVP段階では過剰な設計を避けてください。

## 役割分担

- 実装には AI を活用します。
- 要件定義、設計判断、レビュー、修正指示の責任は開発者本人が持ちます。
- 変更提案時は、「何を作るか」と「何を作らないか」を明確にしてください。
- 変更が大きくなる場合は、一度にまとめて進めず、レビューしやすい単位に分割してください。

## データ取得・スクレイピング方針

- 対象は環境省の報道発表一覧ページです。
- 初回取得では、公開済みデータを可能な範囲で順次取得します。
- 一覧ページのページネーションを考慮してください。
- 重複登録を避ける前提で実装してください。
- 取得停止条件、異常系、形式揺れを明示的に扱ってください。
- サイト構造に依存する処理は、影響範囲が分かるよう責務分割してください。

## API 実装方針

- API は一覧取得、検索、カテゴリ絞り込み、ブックマーク操作を中心に構成してください。
- 一覧取得は **新着順** と **ページネーション** に対応してください。
- API レスポンスの形式は一貫性を保ってください。
- バリデーション、エラーハンドリング、HTTP ステータスの整合性を意識してください。

## フロントエンド実装方針

- 一覧表示、検索、カテゴリ絞り込み、ブックマークを MVP の中心機能とします。
- UI 実装では、見た目よりもまず振る舞いの正しさを優先してください。
- ローディング、エラー表示、空データ時の表示を考慮してください。
- FastAPI をデータ取得元とし、責務をフロントエンド側に寄せすぎないでください。

## テスト方針

- すべてを厳密な TDD で進めるのではなく、重要なロジックではテスト先行を重視してください。
- 特に以下は、可能な限りテストを先に検討してください。
  - 重複判定
  - カテゴリ分類ロジック
  - 検索・ページネーションを含む API の挙動
  - ブックマーク追加・削除・一覧取得
- スクレイピング初期実装や UI の見た目調整では、まず動作確認を優先して構いません。
- テストを追加する場合は、期待する振る舞いが分かる粒度で記述してください。

## テストと検証

- テスト、lint、typecheck がある場合は、変更後に実行すべきコマンドを明示してください。
- 未整備のコマンドについては、存在しない前提で断定せず、必要なら追加提案とセットで示してください。
- 変更完了時には、最低限の動作確認項目を列挙してください。
- 実装詳細ではなく、ユーザーから見た振る舞いを重視してください。

## Git運用

- `main` / `master` への直接コミットは避けてください。
- コミットメッセージを提案する場合は、コンベンショナルコミット形式を優先してください。
- コミットはできるだけ原子的に保ってください。
- レビューしやすい差分単位を意識してください。

## 依存関係

- 本当に必要な依存関係のみ追加してください。
- 新しい依存関係を追加する場合は、追加理由を簡潔に説明してください。
- lockfile や依存定義ファイルは、必要に応じて更新してください。
- 依存関係の追加で代替可能な標準機能がある場合は、まずそちらを検討してください。

## ドキュメント

- README や関連ドキュメントが変更の影響を受ける場合は、更新の必要性を指摘してください。
- 重要な設計判断がある場合は、後から追える形で記録することを意識してください。
- 以下のドキュメントとの整合性を意識してください。
  - `docs/project-overview.md`
  - `docs/requirements.md`
  - `docs/tasks.md`
  - `docs/tech-stack.md`

## コマンド

### フロントエンド

- インストール: `pnpm install`
- 開発サーバー: `pnpm dev`
- lint: `pnpm lint`
- typecheck: `pnpm typecheck`（存在する場合）
- test: `pnpm test`（存在する場合）

### Python / バックエンド / スクレイパー

- 依存管理: `uv sync`
- 開発サーバー: FastAPI 側の起動コマンド定義に従ってください。
- test: `pytest`（存在する場合）

### Docker

- `docker compose` を用いたローカル起動を基本とします。
